<task type="checkpoint:human-action">
  <name>Task 1: Install cloudflared via Cloudflare official apt repository</name>
  <read_first>No repo file — this is a host-side installation step. Verify cloudflared is not already installed before running: `cloudflared --version`</read_first>
  <what-built>N/A — this step installs the cloudflared binary on the host. There are no repository file changes.</what-built>
  <how-to-verify>
Run the following commands in order on the host machine (Debian/apt-based Linux):

**Step 1 — Check if cloudflared is already installed:**
```bash
cloudflared --version
```
If this returns a version string (e.g., `cloudflared version 2024.x.x`), skip to Task 2.

**Step 2 — Add Cloudflare GPG key:**
```bash
sudo mkdir -p --mode=0755 /usr/share/keyrings
curl -fsSL https://pkg.cloudflare.com/cloudflare-main.gpg | sudo tee /usr/share/keyrings/cloudflare-main.gpg >/dev/null
```

**Step 3 — Add Cloudflare apt repository:**
```bash
echo 'deb [signed-by=/usr/share/keyrings/cloudflare-main.gpg] https://pkg.cloudflare.com/cloudflared any main' | sudo tee /etc/apt/sources.list.d/cloudflared.list
```

**Step 4 — Install cloudflared:**
```bash
sudo apt-get update && sudo apt-get install cloudflared
```

**Step 5 — Confirm installation:**
```bash
cloudflared --version
```
Expected output: a version string such as `cloudflared version 2024.x.x (built YYYY-MM-DD ...)`

**Confirm:** Type "installed" and paste the version output, or describe any error.
  </how-to-verify>
  <acceptance_criteria>
    - `cloudflared --version` returns a version string (non-empty, no "command not found")
    - `/usr/bin/cloudflared` exists: `ls -la /usr/bin/cloudflared` returns a file entry
    - `/etc/apt/sources.list.d/cloudflared.list` exists: `cat /etc/apt/sources.list.d/cloudflared.list` shows the Cloudflare repo entry
  </acceptance_criteria>
  <resume-signal>Type "installed" and paste the output of `cloudflared --version`</resume-signal>
</task>

<task type="checkpoint:human-action">
  <name>Task 2: Obtain tunnel token from Cloudflare dashboard and register cloudflared systemd service</name>
  <read_first>No repo file. Prerequisite: Task 1 must be complete (cloudflared binary installed).</read_first>
  <what-built>N/A — this step registers cloudflared as a persistent systemd service. No repository files change.</what-built>
  <how-to-verify>
**Step 1 — Obtain the tunnel token from Cloudflare Zero Trust dashboard:**

1. Log in to https://one.dash.cloudflare.com
2. Go to Networks → Tunnels
3. Find the tunnel for `sontocodemo.sharrief.com` (or create a new one if none exists)
4. Click the tunnel → click "Configure" → click "Install and run a connector"
5. Select "Debian" and copy the token value shown in the `cloudflared service install` command

The token is a long base64-like string (starts with `eyJ...`). Copy only the token value, not the full command.

**IMPORTANT — Token security:**
- Do NOT paste the token into any file, .env, or shell script
- Pass it directly in the terminal command below
- If your shell saves history, consider prefixing the command with a space to suppress history logging (bash: `HISTCONTROL=ignorespace`, then run the command with a leading space)

**Step 2 — Register the tunnel as a systemd service:**

Replace `<YOUR_TUNNEL_TOKEN>` with the token copied from the dashboard:

```bash
sudo cloudflared service install <YOUR_TUNNEL_TOKEN>
```

Expected output: Messages indicating the cloudflared service was installed and started (e.g., "Installing cloudflared service", "cloudflared service installed successfully").

**Step 3 — Confirm the service is running:**
```bash
sudo systemctl status cloudflared
```
Expected: `Active: active (running)` in the output.

**Step 4 — Confirm the service starts on boot:**
```bash
sudo systemctl is-enabled cloudflared
```
Expected output: `enabled`

**Step 5 — Check service logs for tunnel connection:**
```bash
sudo journalctl -u cloudflared --no-pager -n 20
```
Look for a line containing "Connection registered" or "Registered tunnel connection" — this confirms the tunnel is connected to the Cloudflare edge.

**Confirm:** Type "running" and paste the output of `sudo systemctl status cloudflared`.
  </how-to-verify>
  <acceptance_criteria>
    - `sudo systemctl status cloudflared` output contains `Active: active (running)`
    - `sudo systemctl is-enabled cloudflared` returns `enabled`
    - `sudo journalctl -u cloudflared --no-pager -n 20` contains "registered" or "connected" (case-insensitive)
    - `/etc/systemd/system/cloudflared.service` exists: `ls /etc/systemd/system/cloudflared.service` returns a file
    - No file in the repository contains the tunnel token string
  </acceptance_criteria>
  <resume-signal>Type "running" and paste the output of `sudo systemctl status cloudflared`</resume-signal>
</task>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cloudflare edge → host | Cloudflare Tunnel terminates public TLS; cloudflared on host connects outbound to Cloudflare — no inbound firewall ports needed |
| host → Docker container | Port 8901 on localhost bridges to container port 8080; only processes on the host can reach it (no public 0.0.0.0 binding) |
| user → tunnel token | Token is a secret credential granting control of the tunnel; mishandled storage creates privilege escalation risk |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Information Disclosure | Tunnel token | mitigate | Token passed interactively to `cloudflared service install`; plan explicitly prohibits writing token to any file; user instructed to use HISTCONTROL=ignorespace to suppress shell history |
| T-02-05 | Elevation of Privilege | cloudflared systemd service | accept | Service runs as root (required by `cloudflared service install`); this is the standard cloudflared deployment model; no privileged container access granted |
| T-02-06 | Spoofing | https://sontocodemo.sharrief.com DNS | accept | Cloudflare manages DNS and TLS certificate; subdomain is proxied (orange cloud); CNAME ownership is verified by Cloudflare at tunnel creation time |
| T-02-07 | Denial of Service | Public URL reachability | accept | Cloudflare provides DDoS protection at the edge for proxied subdomains; no additional mitigation needed at this scope |
| T-02-08 | Tampering | docker-compose.yml port change | accept | Port 8080 replaced by 8901 on host only; container-internal port unchanged; existing healthcheck and depends_on remain intact |
</threat_model>
